
# 9.4 SQL innjekcija

[Sadržaj](_00.0-sr.md)

## Šta je SQL injekcija

Napadi SQL injekcijom su (kao što i samo ime sugeriše) jedan od mnogih tipova napada injekcijom skripti. U veb razvoju, ovo su najčešći oblici bezbednosnih ranjivosti. Napadači ih mogu koristiti za dobijanje osetljivih informacija iz baza podataka, a aspekti napada mogu uključivati dodavanje korisnika u bazu podataka, izvoz privatnih datoteka, pa čak i dobijanje najviših sistemskih privilegija za sopstvene zlonamerne svrhe.

SQL injekcija se dešava kada veb aplikacije ne filtriraju efikasno korisnički unos, ostavljajući vrata širom otvorena napadačima da pošalju zlonamerni SQL kod upita serveru. Aplikacije često primaju ubrizgani kod kao deo napadačevog unosa, što na neki način menja logiku originalnog upita. Kada aplikacija pokuša da izvrši upit, umesto toga se izvršava zlonamerni kod napadača.

## Primeri SQL injekcija

Mnogi veb programeri ne shvataju kako se SQL upiti mogu menjati i mogu pogrešno verovati da su to pouzdane komande. Kao što svi znaju, SQL upiti mogu da zaobiđu kontrole pristupa, čime zaobilaze standardne provere autentifikacije i autorizacije. Štaviše, moguće je pokretati SQL upite pomoću komandi na nivou host sistema.

Hajde da pogledamo neke stvarne primere kako bismo detaljno objasnili proces SQL injekcije.

Razmotrite sledeći jednostavan obrazac za prijavu:

> ...  
> Korisničko ime: _____  
> Lozinka      : _____  
> ...

Naša obrada formulara može izgledati ovako:

```go
user_name := r.Form.Get(“user_name”) 
password := r.Form.Get(“password”) 
```

```sql
sql := “SELECT * FROM user WHERE user_name='” + user_name + “' AND password='” + password + “'”
```

Ako korisnik unese korisničko ime ili lozinku kao: 'myuser' or 'foo' = 'foo'

Tada naš SQL postaje sledeći:

```sql
SELECT * FROM user WHERE username='myuser' or 'foo' = 'foo' -- AND password='xxx'
```

U SQL-u, sve posle -- je komentar. Stoga, umetanje -- kao što je napadač uradio gore, menja upit na fatalan način, omogućavajući napadaču da se uspešno prijavi kao korisnik bez važeće lozinke.

Mnogo opasniji eksploiti postoje za MSSQL SQL injekcije, a neki čak mogu da izvršavaju sistemske komande. Sledeći primeri će pokazati koliko SQL injekcije mogu biti strašne u nekim verzijama MSSQL baza podataka.

```sql
sql := "SELECT * FROM products WHERE name LIKE '%' + prod + „%'" Db.Exec(sql)
```

Ako napadač pošalje %' exec master..xp_cmdshell ' net user test testpass /ADD' -- kao promenljivu "prod", onda će sql postati

sql := "SELECCT * IZ products WHERE name LIKE '%a%' exec master…xp_cmdshell 'net user test testpass /ADD'–%'“

MSSQL server izvršava SQL naredbu koja uključuje komande u promenljivoj "prod" koju je naveo korisnik, što dodaje nove korisnike sistemu. Ako se ovaj program pokrene onakav kakav jeste i MSSQLSERVER servis ima dovoljna prava, napadač može da registruje sistemski nalog za pristup ovoj mašini.

Iako su gore navedeni primeri vezani za određeni sistem baze podataka, to ne znači da drugi sistemi baza podataka ne mogu biti izloženi sličnim vrstama napada. Principi iza SQL injekcija napada ostaju isti, iako se metoda kojom se izvršavaju može razlikovati.

## Kako sprečiti SQL injekciju

Možda mislite da bi napadač morao da zna informacije o strukturi ciljne baze podataka da bi izvršio napad SQL injekcijom. Iako je to tačno, teško je garantovati da napadač neće moći da pronađe ove informacije i kada ih dobije, baza podataka može biti ugrožena. Ako koristite softver otvorenog koda za pristup bazi podataka, kao što je aplikacija foruma, uljezi mogu lako da dođu do povezanog koda. Očigledno je da su, uz loše dizajniran kod, bezbednosni rizici još veći. Discus, phpwind i phpcms su neki primeri popularnih programa otvorenog koda koji su bili ranjivi na napade SQL injekcijom.

Ovi napadi se dešavaju na sistemima gde mere predostrožnosti nisu prioritet. Već smo to rekli, reći ćemo ponovo: nikada ne verujte bilo kakvoj vrsti unosa, posebno korisničkim podacima. Ovo uključuje podatke koji dolaze iz polja za izbor, skrivenih polja za unos ili kolačića. Kao što je pokazao naš prvi primer iznad, čak i navodno normalni upiti mogu izazvati katastrofe.

Napadi SQL injekcijama mogu biti razarajući - kako uopšte možemo početi da se branimo od njih?

Sledeći predlozi su dobra polazna tačka za sprečavanje SQL injekcija:

- Strogo ograničite dozvole za operacije baze podataka tako da korisnici imaju samo minimalni skup dozvola potrebnih za obavljanje svog posla, čime se minimizira rizik od napada ubrizgavanjem u bazu podataka.
- Proverite da li ulazni podaci imaju očekivani format podataka i strogo ograničite tipove promenljivih koje se mogu poslati. Ovo može uključivati podudaranje regularnih izraza ili korišćenje paketa `strconv` za konvertovanje stringova u druge osnovne tipove radi čišćenja i procene.
- Transkodirajte ili izbegavajte parove specijalnih karaktera ('"&*; itd.) pre nego što ih sačuvate u bazi podataka. Go `text/template` paket ima `HTMLEscapeString` funkciju koja se može koristiti za vraćanje izbegnutog HTML-a.
- Koristite parametrizovani interfejs za upite vaše baze podataka. Parametrizovani iskazi koriste parametre umesto spajanja promenljivih korisničkog unosa u ugrađenim SQL iskazima; drugim rečima, oni direktno ne spajaju SQL iskaze. Na primer, koristeći funkciju `Prepare` iz Go `database/sql` paketa, možemo kreirati pripremljene iskaze za kasnije izvršavanje sa `Query` ili `Exec(query string, args... interface {})`.
- Pre nego što objavite svoju aplikaciju, temeljno je testirajte koristeći profesionalne alate za otkrivanje ranjivosti SQL injekcija i njihovo popravljanje, ako postoje. Postoji mnogo onlajn alata otvorenog koda koji rade upravo to, kao što su sqlmap, SQLnija, da nabrojimo samo neke.
- Izbegavajte štampanje informacija o SQL greškama na javnim veb stranicama. Napadači mogu da koriste ove poruke o greškama za izvođenje SQL injekcija napada. Primeri takvih grešaka su greške u tipu, greške u poljima koja se ne podudaraju ili bilo koje greške koje sadrže SQL izraze.

## Rezime

Kroz gore navedene primere, saznali smo da je SQL injekcija veoma realna i veoma opasna ranjivost veb bezbednosti. Kada pišemo veb aplikacije, trebalo bi da obratimo pažnju na svaki mali detalj i da se bezbednosnim problemima bavimo sa najvećom pažnjom. To će dovesti do boljih i bezbednijih veb aplikacija i na kraju može biti odlučujući faktor u tome da li će vaša aplikacija biti uspešna ili ne.
