
# 11.1 Obrada grešaka

[Sadržaj](_00.0-sr.md)

Glavna razmatranja dizajna u Gou su zasnovana na sledećim idejama:

- jednostavna, jasna i koncizna sintaksa (slično C-u) i
- iskazi koji su eksplicitni i ne sadrže nikakve skrivene ili neočekivane stvari.

Goova šema za rukovanje greškama odražava sve ove principe u načinu na koji je implementirana. Ako ste upoznati sa C jezikom, znaćete da je uobičajeno vraćati vrednosti -1 ili NULL da bi se naznačilo da je došlo do greške.

Međutim, korisnici koji nisu upoznati sa C API-jem neće tačno znati šta ove povratne vrednosti znače. U C-u nije eksplicitno da li vrednost `0` ukazuje na uspeh ili neuspeh.

S druge strane, Go eksplicitno definiše tip koji se naziva `error` isključivo sa svrhom izražavanja grešaka. Kad god se funkcija vrati, proveravamo da li je promenljiva greške `nil` ili ne, kako bismo utvrdili da li je operacija bila uspešna. Na primer, ako funkcija `os.Open` ne uspe, vratiće promenljivu greške koja nije `nil`.

```go
func Open(name string) (file * File, err error)
```

Evo primera kako bismo postupili sa greškom u `os.Open`. Prvo, pokušavamo da otvorimo datoteku. Kada se funkcija vrati, proveravamo da li je uspela ili ne tako što upoređujemo povratnu vrednost greške sa `nil`, pozivajući funkciju `log.Fatal` da ispiše poruku o grešci ako je vrednost različita od nil:

```go
f, err := os.Open("filename.ext")
if err != nil {
  log.Fatal(err)
}
```

Slično funkciji `os.Open`, funkcije u standardnim Go paketima vraćaju promenljive greške kako bi olakšale obradu grešaka. Ovaj odeljak će detaljno objasniti dizajn tipova grešaka i razmotriti kako pravilno obraditi greške u veb aplikacijama.

## Tip greške

`error` je tip interfejsa sa sledećom definicijom:

```go
type error interface {
    Error() string
}
```

`error` je ugrađeni tip interfejsa. Njegovu definiciju možemo pronaći u ugrađenom paketu ispod. Takođe imamo mnogo internih paketa koji koriste `error` privatnu strukturu pod nazivom `errorString`, koja implementira interfejs za greške:

```go
// errorString is a trivial implementation of error.
type errorString struct {
    s string
}

func (e *errorString) Error() string {
    return e.s
}
```

Možete konvertovati običan string u string `errorString` tipa "through" `errors.New` da biste dobili objekat koji zadovoljava interfejs greške. Njegova interna implementacija je sledeća:

```go
// New returns an error that formats as the given text.
func New(text string) error {
    return &errorString{text}
}
```

Sledeći primer pokazuje kako se koristi `errors.New`:

```go
func Sqrt(f float64) (float64, error) {
    if f < 0 {
        return 0, errors.New("math: square root of negative number")
    }
    // implementation
}
```

U sledećem primeru, našoj funkciji prosleđujemo negativan broj `Sqrt`. Proverom `err` promenljive, proveravamo da li je objekat greške različit od `nil` koristeći jednostavno poređenje sa `nil`. Rezultat poređenja je tačan, pa se poziva metoda `fmt.Println` ( `fmt` paket poziva metodu greške kada se bavi pozivima grešaka ) da bi se ispisala greška.

```go
f, err := Sqrt(-1)
if err != nil {
    fmt.Println(err)
}
```

## Prilagođene greške

Kroz gornji opis, znamo da je `Error` Go interfejs. Definisanjem strukture koja implementira ovaj interfejs, možemo implementirati njihove definicije grešaka. Evo primera iz JSON paketa:

```go
type SyntaxError struct {
    msg string // error description
    Offset int64 // where the error occurred
}

func (e * SyntaxError) Error() string {return e.msg}
```

Polje greške `Offset` neće biti ispisano tokom izvršavanja programa kada se pojave sintaksičke greške, ali korišćenjem tvrdnje tipa, možete ispisati željenu poruku o grešci:

```go
if err := dec.Decode(&val); err != nil {
    if serr, ok := err.(*json.SyntaxError); ok {
        line, col := findLine(f, serr.Offset)
        return fmt.Errorf("%s:%d:%d: %v", f.Name(), line, col, err)
    }
    return err
}
```

Treba napomenuti da kada funkcija vrati prilagođenu grešku, povratna vrednost je podešena na preporučeni tip greške, a ne na prilagođeni tip greške. Vodite računa da ne deklarišete unapred promenljive prilagođenih tipova grešaka. Na primer:

```go
func Decode() *SyntaxError {
    // error, which may lead to the caller's err != nil comparison to always be true.
    var err * SyntaxError // pre-declare error variable
    if an error condition {
        err = &SyntaxError{}
    }
    return err // error, err always equal non-nil, causes caller's err != nil comparison to always be true
}
```

Pogledajte <http://golang.org/doc/faq#nil_error> za detaljnije objašnjenje.

Gore navedeni primer pokazuje kako se implementira jednostavan prilagođeni tip greške. Ali šta ako nam je potrebno sofisticiranije rukovanje greškama? U ovom slučaju moramo se pozvati na pristup  `net` paketa:

```go
package net

type Error interface {
    error
    Timeout() bool   // Is the error a timeout?
    Temporary() bool // Is the error temporary?
}
```

Koristeći tvrdnju tipa, možemo proveriti da li je naša greška tipa `net.Error`, kao što je prikazano u sledećem primeru. Ovo nam omogućava da preciziramo našu obradu grešaka - ako se na mreži dogodi privremena greška, ona će se uspavati 1 sekundu, a zatim će ponovo pokušati operaciju.

```go
if nerr, ok := err.(net.Error); ok && nerr.Temporary() {
    time.Sleep(1e9)
    continue
}
if err != nil {
    log.Fatal(err)
}
```

## Obrada grešaka

Go obrađuje greške i proverava povratne vrednosti funkcija na način sličan C-u, što se razlikuje od načina na koji to radi većina drugih glavnih jezika. Ovo čini kod eksplicitnijim i predvidljivijim, ali i detaljnijim. Da bismo smanjili redundantnost našeg koda za obradu grešaka, možemo koristiti apstraktne funkcije za obradu grešaka koje nam omogućavaju da implementiramo slično ponašanje pri obradi grešaka:

```go
func init() {
    http.HandleFunc("/view", viewRecord)
}

func viewRecord(w http.ResponseWriter, r *http.Request) {
    c := appengine.NewContext(r)
    key := datastore.NewKey(c, "Record", r.FormValue("id"), 0, nil)
    record := new(Record)
    if err := datastore.Get(c, key, record); err != nil {
        http.Error(w, err.Error(), 500)
        return
    }
    if err := viewTemplate.Execute(w, record); err != nil {
        http.Error(w, err.Error(), 500)
    }
}
```

Gore navedeni primer pokazuje kako je pristup podacima i poziv šablona otkrio grešku. Kada dođe do greške, poziv ujedinjene funkcije za rukovanje http.Error vraća klijentu kod greške 500 i prikazuje odgovarajuće podatke o grešci. Ali kako se vrši sve više i više poziva HandleFunc, povećava se i logički kod za rukovanje greškama. Možemo prilagoditi ruter da smanji kod (pogledajte treće poglavlje HTTP-a za više detalja).

```go
type appHandler func(http.ResponseWriter, *http.Request) error

func (fn appHandler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    if err := fn(w, r); err != nil {
        http.Error(w, err.Error(), 500)
    }
}
```

Iznad smo definisali prilagođeni ruter. Zatim možemo registrovati naš hendler kao i obično:

```go
func init() {
    http.Handle("/view", appHandler(viewRecord))
}
```

Rukovalac `/view` se zatim može obraditi sledećim kodom; mnogo je jednostavnije od naše originalne implementacije, zar ne?

```go
func viewRecord(w http.ResponseWriter, r *http.Request) error {
    c := appengine.NewContext(r)
    key := datastore.NewKey(c, "Record", r.FormValue("id"), 0, nil)
    record := new(Record)
    if err := datastore.Get(c, key, record); err != nil {
        return err
    }
    return viewTemplate.Execute(w, record)
}
```

Primer obrađivača grešaka iznad će vratiti korisnicima kod interne greške 500 kada se dogode bilo kakve greške, pored ispisa odgovarajućeg koda greške. U stvari, možemo prilagoditi tip vraćene greške kako bismo prikazali poruku o grešci koja je prijateljskija za programere sa informacijama koje su korisne za otklanjanje grešaka, kao što je prikazano:

```go
type appError struct {
    Error   error
    Message string
    Code    int
}
```

Naš prilagođeni ruter se može promeniti shodno tome:

```go
type appHandler func(http.ResponseWriter, *http.Request) *appError

func (fn appHandler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    if e := fn(w, r); e != nil { // e is *appError, not os.Error.
        c := appengine.NewContext(r)
        c.Errorf("%v", e.Error)
        http.Error(w, e.Message, e.Code)
    }
}
```

Nakon što završimo sa modifikovanjem naše prilagođene greške, naša logika se može promeniti na sledeći način:

```go
func viewRecord(w http.ResponseWriter, r *http.Request) *appError {
    c := appengine.NewContext(r)
    key := datastore.NewKey(c, "Record", r.FormValue("id"), 0, nil)
    record := new(Record)
    if err := datastore.Get(c, key, record); err != nil {
        return &appError{err, "Record not found", 404}
    }
    if err := viewTemplate.Execute(w, record); err != nil {
        return &appError{err, "Can't display record", 500}
    }
    return nil
}
```

Kao što je gore prikazano, možemo vratiti različite kodove grešaka i poruke o grešci u našim prikazima, u zavisnosti od situacije. Iako ova verzija našeg koda funkcioniše slično prethodnoj verziji, ona je eksplicitnija, a njeni upiti za poruke o grešci su razumljiviji. Svi ovi faktori mogu pomoći da vaša aplikacija bude skalabilnija kako se složenost povećava.

## Rezime

Tolerancija na greške je veoma važan aspekt svakog programskog jezika. U jeziku Go, to se postiže kroz rukovanje greškama. Iako JE `Error` samo jedan interfejs, može imati mnogo varijacija u načinu na koji je implementiran, i možemo ga prilagoditi našim potrebama od slučaja do slučaja.

Predstavljanjem ovih različitih koncepata rukovanja greškama, nadamo se da ste stekli uvid u to kako da implementirate bolje šeme rukovanja greškama u svojim veb aplikacijama.
